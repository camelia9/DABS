Abstract

DABS - Database Smart Web Tool is a Web tool able to compare, run, visualize, and recommend various database management systems (DBMS) according to developer preferences and specific usages – e.g., implementing financial desktop applications, e-commerce Web applications, educational platforms, document management systems or others.

We wish to develop a smart platform which will be able to provide an innovative mean to find the most suitable solutions for project developing, in order to be up to date with the cutting edge database technologies. Our secondary goal is to facilitate the process through which our users can learn new skills regarding databases usage methods. All of the provided functionalities are presented in a user-friendly maner through a web interface, whilst offerint a powerful tool not only for experienced developers but also for rookies.

Architecture

In the first section we briefly described the main goals of the application. The architecture has two main components: a front-end part, for the client interaction, built over Angular Material JS framework and a back-end component implemented in NodeJs, designed following a few well known paradigms and hosted by using various resources provided by the AWS Cloud Platform. These two components will communicate over the HTTP protocol, using the JSON and its extension JSON-LD as the established data exchange format. The client will build JSON requests to send requests toward the backend component, while the received responses will be in the JSON-LD format. In the frontend implementation, we'll have a module for JSON-LD to HTML+RDFa, in order to have semantic data encoded right in our single page application.

The web services conglomerate provides a variety of functionalities, such as getting a news feed list containg the latest news in the world of databases, finding the most suitable datase for certain kinds of projects, a playground for multiple database solutions and the visualization of our user's preferences relatively to our apps proposed technologies. An advantage to the proposed architecture is that each and every functionality is independent to the other, therefore our application will not be fully affected if a downtime were to occur in one of the given components. More details about each component's particular behaviour can be seen in the corresponding sections from below.

Back-end Component

The Back-end Component Architecture is built over two main paradigms: 'REST' and 'Serverless'.

  • REST or REpresentational State Transfer, is an architectural style for providing standards between computer systems on the web, making it easier for systems to communicate with each other. REST-compliant systems, often called RESTful systems, are characterized by how they are stateless and separate the concerns of client and server.

  • Serverless computing (or serverless for short), is an execution model where the cloud provider (AWS, Azure, or Google Cloud) is responsible for executing a piece of code by dynamically allocating the resources. And only charging for the amount of resources used to run the code. The code is typically run inside stateless containers that can be triggered by a variety of events including http requests, database events, queuing services, monitoring alerts, file uploads, scheduled events (cron jobs), etc. The code that is sent to the cloud provider for execution is usually in the form of a function. Hence serverless is sometimes referred to as “Functions as a Service” or “FaaS”.

This module is composed of the following components:

  • AWS Api Gateway - Its role is to aggregate the lambda functions cluster and to secure a unique point of access to all functionalities for the web client to call. From the gateway, the request can go two ways: either to the authentication lambda (if the user is not logged in, therefore he does not have a JWT token) or to the authorization lambda (if the user has a JWT token and therefore is already authenticated).
  • Authentication Lambda - Its purpose is to fullfil the process of logging in for users that want to use our services. In the SignIn View of our application, the user can select either to login with one of his social media accounts or to log in with an account from our system. This option is sent towards this lambda, which will verify the provided details either by consulting an API from Google, Facebook or Twitter or by searching the user + hash password combination using the SparQL Query Runner Endpoint on the users dabatase from our GraphDB instance. If the login process is successful, a JWT will be returned to user in the frontend application, which will be stored in cookies for further usage when sending the requests. If the login process fails, a HTTP specific error code will be returned alongside with an error message.
  • Authorization Lambda - This service provides the authorization step that allows the user access to the functionalities enclosed in the other lambdas. When a request is received through the API Gateway, it will first arrive here, where the provided JWT token is validated by searching if there is an active session associated to the IP + JWT + user_id combination. If the token does not exist or it is expired, an error response is sent back to the user. If the authorization is successfull, the request is forwarded to the lambda that handles the requested functionality.
  • DynamoDB Host - In this host we have a database for the active user sessions, which will contain the IP + JWT + user_id pairs; This will be used in the Authentication Lambda to insert an active session for the logged in user and will be further consulted by the Authorization Lambda to validate the authenticity of the JWT token.
  • User Lambda - This lambda exposes functionalities for the user entity manipulations, such as adding an user to the RDF data store, updating one or deleting one.
  • Preferences Lambda - This lambda exposes functionalities for the user's preferences entity manipulations, such as adding a preference to the RDF data store, updating one or deleting one. It is also used for retrieving all preferences. The responses from this endpoint are in JSON-LD format for entity related functionalities.
  • Database Lambda - This lambda exposes functionalities for the databse entity manipulations, such as adding a new database to the RDF data store, updating one or deleting one. It is also used for retrieving all databases. Furthermore, this lambda is used for gaining access to the database playground cluster that allows the user to try different technologies, such as Mongo, Redis and PostgreSQL. The responses from this endpoint are in JSON-LD format for entity related functionalities, while the others are in plain JSON.
  • News Feed Lambda - This lambda allows the retrieval of the latest news about databases. It uses DbWeekly as a data source, which can be integrated in the application using tefter.io. The responses from this endpoint are in plain JSON.
  • SparQL Query Runner Endpoint - This component is in charge of transforming the received parameters in lambdas (through requests) in various SparQL queries, depending on which data source we need to interogate. Once the response is received in the lambda from GraphDB, it will be sent as JSON-LD back to the client.
  • Database Playground - It is composed of one Redis instance, one PostgreSQL instance and one MongoDB instance, all deployed in AWS, eventually on a dedicated VM. They will be queried directly by the Database Lambda when someone sends a query.
  • GraphDB Host - It is used to store the RDF data for our three main entities: Users, Databases and Preferences. It will be queried from the SparQL Query Runner Endpoint.

As a summary, the Lambda functions offer the main functionalities, being the core elements of our architecture. They are aggregated via an API Gateway, which oferrs an unique point of access for the web client. Some of the lambda functions communicate with data sources represented as ontologies for databases, users and preferences stored in a RDF repository (GraphDB). The query language for accessing the datasources is SparQL and the communication contract established between the client and the services provider is a mix between JSON (used for request) and JSON-LD (sent through responses). The resource access is made securely via a midway security lambda that ensures all the API calls have to be from logged users. The way to interact with the API can be seen in the API specification from our SwaggerHub Project.

Front-end Component

The user interface will be created using the AngularJS framework combined with Material Design, which is ideal for building single-page applications. AngularJS Material is an implementation of Google's Material Design Specification (2014-2017). This project provides a set of reusable, well-tested, and accessible UI components for AngularJS developers. The web interface will be hosted in AWS S3 Bucket, in order to communicate easily with the other components of the project. The data which is used to populate the web page in specific context is received either in the JSON format or its extension JSON-LD. For the latter, we'll use a parser to transform it in HTML+RDFa format, which will be loaded into the page using custom built AngularJS directives.

In the following figure, right side, we can see the sign in page, which will be the first view an user sees. There are two main options to authenticate in the application: either login with an account created in our application or log in with a social media account. The log-in step is necessary because we want to be able to gather data about the user preferences in order to build some statistics on various periods of time and to know which technologies may be suited for each and every user. If the user does not have an account, nor does he want to log in with social media, there is a create account section, which can be seen in the left side of the picture. It requires some simple information, such as email, password and name. Once the user authenticates successfully in our application a JWT token is generated for the current session. The JWT should be passed along with each subsequent request in order to fullfil the auhtorization process for the backend functionalities.

In the next figure, we can see the Home View, where we first present the current trends in database searches for the past week. This graphic's data is computed using our user's past queries and preferences and the technology recommendations they resulted in (gathered from his inputs in the Recommendation Wizard). The second chart analyses the our users past queries based on different criteria such as searches paradigms, running environments, types of databases and so on ans so forth. It should highlight the most searched values in order to help the user gain insight over what is trending right now. The second part of this page is composed of a newsfeed with the most recent information gathered from DBWeekly. The user is able to share this information through his social media account (if he is logged in with one) or send it through email (if he is logged in with an application specific account). There is also an option to view the original source, should he want to look into similar older articles. The main navigation menu is available in the upper right side of the page. It allows us to switch between the dynamic views of our single page application. In the left upper corner, there is a small menu which allows the user to log out of the application or switch to a view with his accounts data and settings.

In the Recommendation Tool View the user is able to select values for the various filtering criteria available. Once the values have been selected, the user can submit his search. Below he can see the top three search results that match his criteria, alongside with the available RDFa information and a knowledge graph (for each result) that ties each search result to other types of databases that may be similar in some ways. In the Database Playground module, the user is able to select a database type from one of the three available options: MongoDB (Document Based), Redis( Key/Value Store) and PostgreSQL. After selecting a database, a shell will be spawned, which will allow the user to run certain queries on each database. First, he should make the setup for the database (such as creating a database, a table/collection, maybe define a schema) and then he can perform queries. Every operation is sent to the backend component and processed during the user's session. The responses to the operations (received from backend) will be shown to the user in this shell. Once the session has ended, the database the user created will be dropped, in order to keep a clean and available playground for all users.

Technologies

  • NodeJS is an open-source, cross-platform, JavaScript runtime environment that executes JavaScript code outside of a browser. Node.js has an event-driven architecture capable of asynchronous I/O. These design choices aim to optimize throughput and scalability in web applications with many input/output operations, as well as for real-time Web applications

  • AngularJS is a structural framework for dynamic web apps. It lets you use HTML as your template language and lets you extend HTML’s syntax to express your application’s components clearly and succinctly. Angular’s data binding and dependency injection eliminate much of the code you currently have to write. And it all happens within the browser, making it an ideal partner with any server technology.

  • AWS (Amazon Web Services) is a secure cloud services platform, offering compute power, database storage, content delivery and other functionality to help businesses scale and grow.

  • GraphDB is an enterprise ready Semantic Graph Database, compliant with W3C Standards. Semantic graph databases (also called RDF triplestores) provide the core infrastructure for solutions where modelling agility, data integration, relationship exploration and cross-enterprise data publishing and consumption are important.

  • JSON-LD is a lightweight Linked Data format. It is based on the JSON format and provides a way to help JSON data interoperate at Web-scale. JSON-LD is an ideal data format for programming environments, REST Web services, and unstructured databases.

  • JWT (JSON Web Tokens) is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object. This information can be verified and trusted because it is digitally signed. JWTs can be signed using a secret (with the HMAC algorithm) or a public/private key pair using RSA.

  • MongoDB is a cross-platform document-oriented database program. Classified as a NoSQL database program, MongoDB uses JSON-like documents with schema. MongoDB is developed by MongoDB Inc. and licensed under the Server Side Public License (SSPL).

  • Redis is an open source (BSD licensed), in-memory data structure store, used as a database, cache and message broker. It supports data structures such as strings, hashes, lists, sets, sorted sets with range queries, bitmaps, hyperloglogs, geospatial indexes with radius queries and streams. Redis has built-in replication, Lua scripting, LRU eviction, transactions and different levels of on-disk persistence, and provides high availability via Redis Sentinel and automatic partitioning with Redis Cluster.

  • PostgreSQL is a powerful, open source object-relational database system with over 30 years of active development that has earned it a strong reputation for reliability, feature robustness, and performance.

  • AWS Lambda lets you run code without provisioning or managing servers. You pay only for the compute time you consume - there is no charge when your code is not running. With Lambda, you can run code for virtually any type of application or backend service - all with zero administration. Just upload your code and Lambda takes care of everything required to run and scale your code with high availability. You can set up your code to automatically trigger from other AWS services or call it directly from any web or mobile app.

  • AWS DynamoDB is a key-value and document database that delivers single-digit millisecond performance at any scale. It's a fully managed, multiregion, multimaster, durable database with built-in security, backup and restore, and in-memory caching for internet-scale applications. DynamoDB can handle more than 10 trillion requests per day and can support peaks of more than 20 million requests per second.

  • AWS API Gateway is a fully managed service that makes it easy for developers to create, publish, maintain, monitor, and secure APIs at any scale. With a few clicks in the AWS Management Console, you can create REST and WebSocket APIs that act as a “front door” for applications to access data, business logic, or functionality from your backend services, such as workloads running on Amazon Elastic Compute Cloud (Amazon EC2), code running on AWS Lambda, any web application, or real-time communication applications. API Gateway handles all the tasks involved in accepting and processing up to hundreds of thousands of concurrent API calls, including traffic management, authorization and access control, monitoring, and API version management.

Conclusions

To sum up, this solution can be a valuable tool for any developer who is in search for trending technologies that are the most suited for his end goal. It also provides an intuitive web UI which allows the user to explore all of the applications features. Furthermore, all functionalities can be queried through an API (without actually needing the client), since we have a decoupled and distributed architecture.

As for the future work, we propose enriching the available data sources for querying technologies, allowing access for a few more technologies in the playground and extending the social media handles integration to the point where users can share between eachother interesting choices of databases and rate the quality of our recommandations in order to improve with each and every query.

Nonetheless, the importance of these kind of approaches in the next web is crucial, as David Amerland emphasizes in Google Semantic Search: Search Engine Optimization (Seo) Techniques That Get Your Company More Traffic, Increase Brand Impact, and Amplify Your Online Presence and hopefully we bring with this project a small contribution to a better future.
